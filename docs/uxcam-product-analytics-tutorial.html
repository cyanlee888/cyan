1. 用量与体验统计 MobileWeb

登录 UXCam 后首页级别的「大盘」，先看趋势再下钻到其他模块。

您可以做的操作

  1. 登录 UXCam，在概览区找到 Usage & UX Statistics（用量与体验统计）。
  2. 在页面中选择分析周期：最近 24 小时 / 7 天 / 30 天。
  3. 在 Usage Statistics 区域，点击各指标卡片（如 Sessions），展开随时间变化的曲线，并与历史周期对比趋势。
  4. UX Statistics 区块基于最近约 2 个月数据计算，用于补充「星期几最活跃、哪块屏停留最久」等体验向信息。

您能得到的结论 / 结果

  • Sessions：选定周期内录到的会话总数。
  • Active Users：该周期内至少打开过一次 App 的用户数。
  • Top events：触发次数最多的行为事件。
  • Top App Version：使用最多的 App 版本（便于发现升级滞后）。
  • Session Duration：平均单次会话时长。
  • Rage Gestures：愤怒点击次数，偏高通常提示界面响应或引导问题。
  • UX 区：最常访问的星期几、停留最长的页面、主要国家/地区、常见设备尺寸等。

→ 官方文档（含图）：Usage & UX Statistics

2. 可定制看板（Customizable Dashboard）MobileWeb

把多个指标放在同一页，支持「报表库一键添加」或「从零搭建」，并可分享、导出、从图表一键跳到会话列表看录屏。

您可以做的操作

A. 新建看板

  1. 进入 Dashboard 区域，点击右上角 Create Dashboard。
  2. 填写名称与描述（方便团队理解用途）。
  3. 设置访问权限：可先设为私有，后续再共享。

B. 添加报表 — 使用报表库（Report Library）

  1. 在看板中点击 Add report。
  2. 按分类浏览（如 Acquisition、Engagement、Events、Issues、Screens、Users 等）或搜索关键词。
  3. 打开预览后点 Add 加入看板；若报表需要选择事件或属性，按页面提示完成配置。
  4. 已添加的预置报表可通过三点菜单 → Edit report修改参数。

C. 从零创建报表（Build from scratch）

  1. 选择数据对象：events、sessions、users、screens，以及移动端还可选 crashes、UI freezes、handled exceptions 等。
  2. 选择指标（metric）、聚合方式（如 total、%、min、max、avg 等，视维度而定）。
  3. 设置分组（Grouping）：默认按时间（天）；也可按设备属性、用户属性等；最多两个分组（官方说明：不支持按小时分组）。
  4. 按需添加 Filters 筛选条件。
  5. 在右侧选择展示形式：Chart / Table / Numeric；选择时间范围后点 Create Report 命名保存。

D. 整理与导出

  • 拖拽调整报表块位置；拖右下角改变大小。
  • 三点菜单：Edit / Rename / 切换图表类型 / Duplicate / Delete / Download（表格、图表可导出 csv、xls、png、jpeg、pdf、svg 等，以界面为准）。
  • 看板级：Single date range 可统一覆盖各报表日期；或保留各报表独立时间。
  • Share Dashboard：团队内共享、或生成公开链接（公开访问者通常不能从图表再点进会话等定性数据，以官方说明为准）。

您能得到的结论 / 结果

  • 一页看清拉新、活跃、转化、崩溃、页面表现等组合指标。
  • 对比不同指标（例如「各平台崩溃数 vs 会话量」）发现异常相关。
  • 从表格/图表一键跳转到会话、事件或用户列表，把「数字」和「真实操作录屏」连起来。
  • 通过公开链接向未开通账号的干系人展示固定看板（权限可控）。

Web 与 Mobile 差异：Web 侧报表数据点侧重 Events、Conversions、Sessions、Screens；Mobile 另含 Issues 等类型，以官方说明为准。

3. 留存分析（Retention Analytics）

定义「首次行为」与「回访行为」，看用户是否在后续周期回来，用于评估新功能、运营活动与长期粘性。

视频：官方在「How to set up a Retention Report」页面提供汇总讲解视频，请在下方官方文档链接打开后于页面内播放（教程内嵌链接可能因附件策略无法直接引用视频源）。

您可以做的操作

  1. 进入 Retention 模块，点击 Create Report。
  2. 选择 Start action（起始行为）：如注册、完成引导、首次访问、某关键事件/页面等；可组合多个起始行为，并可设最少发生次数。
  3. 用 Group by 定义队列粒度：按日/周/月/季将用户分入不同 cohort。
  4. 选择 Return action（回访行为）：代表用户「获得价值」的行为（如下单、观看、生成报告等）；同样支持多个行为及最少次数（多行为时计数规则见官方说明）。
  5. 设置 Return interval：期望用户多久重复一次该行为（日/周/月/季）；并选择Bounded（In）与Unbounded（In or After）留存口径。
  6. 添加 Filters（设备属性、用户属性等）与 Groupings（按版本、国家、订阅类型等拆分对比）。
  7. 保存后结合官方下一篇文档学习阅读留存曲线与留存表，并可保存 Cohort 供其他分析复用。

您能得到的结论 / 结果

  • 各队列在第 1、2、3… 个周期仍执行回访行为的用户比例。
  • 对比不同获客周、不同版本、不同国家的长期留存差异。
  • 判断某次改版或活动后，用户是「短期来一次」还是「持续回来」。

4. 漏斗（Funnels）

把「屏幕 + 事件」串成路径，看每一步流失与转化；支持 AI/规则建议路径，也支持完全自定义。

您可以做的操作

  1. 点击 Create funnel；若不确定步骤，点 Suggestions：先选一个代表最终转化的屏幕或事件，系统会给出 3 条常见路径建议，可预览后 Create。
  2. 手动搭建：选择 This order（严格顺序）或 Any order（无序完成即算）；通过 Add steps 添加屏幕/事件，拖拽排序（最多 15 步）。
  3. 选择统计口径：Session count 或 User count（例如更关心「成交笔数」还是「成交人数」）。
  4. 设置转化窗口：基于会话时，默认须在同一会话内完成所有步骤；基于用户时可设「7 天内」「3 天内」等，用于注册后首单、优惠券有效期等场景。
  5. 可加整漏斗过滤器（国家、平台、是否加购等）与 Group by 对比不同人群转化率。
  6. 命名、选择日期范围，Save funnel；之后可随时编辑步骤或日期。

您能得到的结论 / 结果（分析页）

  • 顶部四项：进入漏斗的会话/用户数、总转化率、总转化次数、平均转化耗时（有序漏斗；两步骤漏斗可分析单段耗时）。
  • 每一步的微转化与流失（drop-off），定位「用户主要死在哪一步」。
  • 按时间、设备、用户属性、事件过滤或分组，评估改版 / A/B 测试 / 新市场投放前后差异。
  • 点击漏斗条带上方（流失）或下方（成功）区域，可直接打开对应会话录屏理解「为什么没转化」或「成功路径长什么样」。

5. 页面与热力图（Screen analytics & Heatmap）

看某一屏上的点击/滑动集中区域、退出率、愤怒点击等；支持框选区域做定量分析，并可跳到会话录屏。

您可以做的操作

  1. 进入 Screens（或 Web 对应屏幕分析入口），选择要分析的页面。
  2. 左侧查看该屏的访问次数、退出率、rage taps、停留时长、手势次数、带崩溃的访问等指标。
  3. 使用过滤器：按日期、用户/设备/会话属性、前后访问路径、是否发生某事件、手势类型与次数等缩小样本。
  4. 在热力图上按下拖拽框选关注区域，查看该区域内的手势数、用户数、首次手势前等待时间等。
  5. 切换到 Screen visits 列表，按下一屏、停留时长等排序，点击播放会话录屏（默认播放该屏切换片段，可继续观看全会话）。

您能得到的结论 / 结果

  • 识别误点高发区、无效点击（unresponsive gestures）、CTA 是否被其他元素分散注意力。
  • 对比改版前后或不同人群（国家、A/B 组）在同一屏上的行为差异。
  • 将热力图发现与漏斗流失步骤交叉验证，形成改版依据。

6. 会话录屏（Session Recording）

回放真实用户操作轨迹；可打标签、写备注、分享给无账号同事。另有 Smart Filters（Beta） 用于智能筛选会话，详见官方文档。

您可以做的操作

  • 在会话播放器中，使用链接图标复制会话 URL，发给未开通账号的同事查看。
  • 使用 Share：填写收件人、留言、选择要分享的片段后发送（由 UXCam 代发邮件）。
  • 在需要长期保留时，通过右上角三点菜单 → Download 下载录像（注意与数据保留策略一致）。
  • 在时间轴上某时刻点 Add note，自动带时间戳；正文中用 @ 提及团队成员（对方需有 UXCam 权限），其会收到邮件并直达对应时间点。
  • 结合官方文档创建 Labels 对会话分类，便于后续筛选与协作。

您能得到的结论 / 结果

  • 把「数字异常」还原为真实操作路径与界面状态，缩短产品/研发定位问题时间。
  • 通过备注与 @，在团队内形成可追踪的讨论记录。
  • 对外分享典型「坏体验 / 好体验」片段，辅助汇报与对齐优先级。

7. 智能事件（Smart Events）

在不写代码的前提下，把「用户对某 UI 元素的手势」或「进入某屏」定义为可统计事件，用于后续漏斗、看板、筛选。

视频：「How to create a Smart Event」官方页面提供分步讲解视频，请从下方链接进入后在页面内观看。

您可以做的操作

  1. 在会话回放右侧活动时间线找到目标手势（单次点击或滑动），展开卡片后点 Create event 或 +。
  2. 为事件命名、写描述，并按需配置 Screen name / UI element name / Element ID 等规则，以一次性覆盖多屏、多同类按钮（详见官方「rules」说明）。
  3. 也可从热力图框选或 Screens → 手势筛选 找到含该元素的会话，再在时间线上创建事件。
  4. 创建后在「Smart Events」管理列表中可编辑、停用或分析趋势（见官方 How to manage / analyze）。

前提条件（官方）：手势须为 Single tap 或 Swipes，且对应 UI 元素在屏幕上可见。

您能得到的结论 / 结果

  • 无需发版即可新增关键行为埋点，加速验证假设。
  • 在事件分析、漏斗、看板中像普通事件一样统计次数、分群、对比。
  • 与 Coded Events 组合：关键路径用代码事件保证精度，长尾探索用 Smart Event 补充（注意官方对 Smart Events 局限的说明）。

8. 编码事件（Coded Events）与事件分析

由研发在 App 内通过 SDK/API 上报的事件与属性，适合做权威业务埋点（订单金额、会员等级等）。

您可以做的操作

  • 在 Events 相关页面按事件名、属性、用户/设备条件筛选与保存常用视图。
  • 阅读官方 Send Coded Events and Properties，与研发对齐事件命名、属性类型与上报时机。
  • 将事件用于漏斗步骤、留存起始/回访动作、看板报表等模块。

您能得到的结论 / 结果

  • 按业务语义精确统计关键业务动作，不受界面改版影响（相对 Smart Event 更稳定）。
  • 与 Smart Event 互补：前者保证「定义清晰」，后者支持「快速探索」。

→ Send Coded Events and Properties  |  → Filter and analyze Smart Events

9. 用户分析（User Analytics / User Properties）

把业务侧用户标签（会员等级、城市、是否付费等）传给 UXCam 后，可在各模块作为过滤器或分组维度。

您可以做的操作

  • 与数据/研发对齐需要上传的 User Properties 列表与取值规范。
  • 在会话列表、热力图、漏斗、留存、看板报表中，选择按这些属性过滤或 Group by。
  • 参考官方「Ideas for custom User Properties」扩展可分析维度。

您能得到的结论 / 结果

  • 看高价值用户 vs 免费用户在同一流程上的行为差异。
  • 按渠道、客群拆分留存与转化，支撑精细化运营决策。

→ Ideas for custom User Properties

10. 问题分析（Crash & UI Freeze）主要 Mobile

集中查看崩溃与界面卡顿（UI Freeze）趋势、影响面、关联会话，便于研发排障与版本优先级决策。

您可以做的操作

  1. 在 Issue Analytics 中先选择关注 Crashes 或 UI Freezes。
  2. 查看概览：问题次数、受影响用户/会话、无问题会话占比等。
  3. 查看趋势与按 App 版本、系统版本、机型 的分布，定位高发环境。
  4. 在列表中点开某一崩溃类型或卡顿类型，阅读日志与统计，并打开关联会话列表逐条回放。
  5. 按需开启 Slack / Microsoft Teams 通知以及时响应（见官方 Issues 通知文档）。

您能得到的结论 / 结果

  • 判断当前版本是否存在回归性崩溃或卡顿峰值。
  • 把「TOP 崩溃签名」与用户实际录屏路径关联，缩短复现时间。
  • 对比不同机型/系统上的体验稳定性，指导测试资源投入。

UXCam 名词解释

控制台与报表中常见英文字段的中文对照、业务含义与统计口径。内容译述自 UXCam 官方帮助中心；若与英文原文不一致，以官方文档为准。

主要参考：UXCam Glossary、 Session FAQs、 UI Freeze's Events。 Web 端部分字段命名可能略有差异，请以当前租户界面为准。

覆盖范围说明

UXCam 控制台与报表中的英文标签数量多、且随 Mobile/Web、套餐与版本迭代；本 Tab 无法保证与当前界面 100% 逐条穷尽。下方已与官方 UXCam Glossary 表格逐条对照补全，并保留漏斗、留存、筛选等分析扩展条目。若遇未收录字段，建议继续在 Dashboard Metrics Explained 等文档中检索。

用户与会话

Session会话

定义：用户在应用内从启动到结束的一次完整访问过程；包含该次访问中的手势、事件、屏幕访问以及用户/设备/会话等元数据。

统计口径：会话在用户打开 App 时开始，在 App进入后台（background）时结束。会话一般在进入后台后上传；用于漏斗、热力图、用户流等分析。

来源：Glossary — Session、 Session FAQs — 起止时间

Video录屏 / 视频

定义：对会话的可视化回放，展示用户在界面上的实际操作与画面变化。

统计口径：会话与视频不同：会话承载交互与元数据；视频是会话的一部分表现。部分会话可能无视频（如达到视频条数上限、条件录制、网络等原因，界面可能显示为「…」）。

来源：Glossary — Video、 Session FAQs — sessions vs videos

Session ID会话 ID

定义：由 UXCam 为每一次会话分配的唯一标识符。

统计口径：一条会话对应一个 Session ID，用于在列表、下钻、分享链接中唯一定位该次访问。

来源：Glossary — Session ID

Active User活跃用户

定义：在选定时间范围内，至少使用过应用一次的独立用户。

统计口径：默认以 Install ID（安装维度设备标识）区分唯一用户；也可通过你们自行设置的 User ID 识别用户。

来源：Glossary — Active User

New User新用户

定义：在选定时间范围内首次访问应用的独立用户。

统计口径：与 Active User 相同，以 Install ID（或你们配置的 User ID）去重，仅统计「第一次出现」的用户。

来源：Glossary — New User

Install ID安装 ID / 设备侧用户标识

定义：UXCam 用于识别「同一安装实例」的标识；在 iOS 上来自系统提供的 Vendor 设备 ID（与发行商绑定的随机 UUID）。

统计口径：作为默认维度统计唯一用户、新用户等；与你们业务账号体系无必然对应关系，除非再通过 User ID 关联。

来源：Glossary — Install ID

Custom user ID / User name自定义用户 ID / 用户名

定义：通过 API（如 UXCam.setUserIdentity）由客户设置的 Custom User ID，用于在控制台识别用户；官方词汇表中 User name 与此同义，并说明亦可指 UXCam User ID（由 UXCam 分配、在界面中展示的「用户名」类标识，以当前控制台为准）。

统计口径：属于用户属性的一种；勿向 Custom user ID 上报 PII（可直接识别个人的敏感信息）。与 Install ID 可同时存在，便于与业务用户对齐。

来源：Glossary — Custom user ID、User name

First seen / Last seen首次出现 / 最近出现

定义：UXCam 记录到该用户最早一次会话的日期；最近一次会话的日期。

统计口径：基于已上传会话的时间戳；用于判断用户生命周期与回访。

来源：Glossary — First seen / Last seen

Session label / User label会话标签 / 用户标签

定义：为会话或用户打上的自定义标签，便于筛选与后续分析。

统计口径：标签为人工或规则赋予的分类维度，不改变会话起止等基本统计。

来源：Glossary — Session label / User label

屏幕与导航

Screens屏幕

定义：在应用中被自动识别的页面/Activity（或 Fragment 等）名称，也可通过 Screen Tagging API 手动指定。

统计口径：一次会话内多次进入同一屏会产生多次「屏幕访问」；热力图、漏斗等按屏幕名聚合。

来源：Glossary — Screens

App quit rate应用退出率（屏幕维度）

定义：在选定日期范围内，用户在某次屏幕访问后直接退出应用的访问占比。

统计口径：以屏幕访问为分母，衡量「离开该屏后是否结束会话（退出 App）」；用于发现易流失页面。

来源：Glossary — App quit rate

Schematic Recording示意图录制（iOS）

定义：在 iOS 上，UXCam 默认用「示意图」方式可视化应用内行为，而非直接录制屏幕位图；通过绘制矩形与文字等方式重建与真实界面布局相符的会话画面。

统计口径：仅适用于 iOS；Android 录一般为图像形式。与隐私、性能策略相关。

来源：Glossary — Schematic Recording

事件与属性

Event事件

定义：用户对应用的交互或状态变化；如点击按钮、完成流程、看到错误提示等。官方在 Event 条目中将其余需追踪的动作统称为以 Custom Events 形式上报；与控制台中的 Coded Event（编码事件）为同一类能力的不同表述。

统计口径：UXCam 默认自动识别两类行为作为事件相关信号：Rage taps 与 UI freezes。其余业务动作须通过代码/API 上报（Coded Event / Custom Events）。

来源：Glossary — Event

Coded Event编码事件 / 埋点事件

定义：通过事件 API 在应用内主动记录的用户交互；不限于手势或进屏，可带自定义属性，灵活描述业务行为。官方说明此类事件在应用侧实现、与平台内创建的 Smart Event 相对。

统计口径：从接入并上报的时刻起开始统计，不会追溯接入前的历史会话。

来源：Glossary — Coded Event

Smart Event智能事件

定义：在 UXCam 控制台基于手势（点击/滑动）或对某屏的访问所创建的事件，用于捕获特定 UI 交互。

统计口径：通常绑定 UI 元素元数据（如 element id、name、type）；可对历史会话回溯应用（与 Coded Event 的「不追溯」不同）。

来源：Glossary — Smart Event

Event Properties事件属性

定义：附属于某次事件上报的键值信息，用于补充上下文（数量、状态、名称等）。

统计口径：官方说明每个事件最多约 20 个属性（以当前文档与套餐为准）。

来源：Glossary — Event Properties

Custom User properties自定义用户属性

定义：除默认字段外，与用户在 UXCam 中档案关联的自定义信息（如会员等级、城市等）。

统计口径：通过 API 发送，官方词汇表提及最多约 100 个附加属性（以官方最新说明为准）；常用于筛选、分组、漏斗切片。

来源：Glossary — Custom User properties

体验信号与稳定性

Rage Tap / Rage Gesture愤怒点击 / 挫败手势

定义：用户在短时间内对屏幕某一区域快速多次点击，通常表示对控件无响应或体验受挫。

统计口径（官方 Glossary）：在屏幕某一半径内，至少连续 3 次点击，且相邻两次点击间隔不超过约 300ms。可在应用设置中对特定屏幕关闭「自动标为愤怒点击」以避免误报（如需要快速连点的控件）。Session FAQ 中亦描述为「多于 3 次」快速点击，实际以官方当前版本说明为准。

来源：Glossary — Rage Tap、 Session FAQs — rage taps

Rage Quits愤怒退出会话

定义：在一次至少发生过愤怒点击的会话中，用户在极短时间内结束会话的行为模式。

统计口径：若会话在至少一次 rage tap 之后的 5 秒内结束，则计为 Rage quit 类会话（具体展示名称以控制台为准）。

来源：Glossary — Rage Quits

Unresponsive Gesture无响应手势

定义：用户做出点击或手势后，应用未产生预期响应；可能是控件缺陷，也可能是用户误点静态文案等非交互元素。

统计口径：作为体验质量信号，常与热力图、屏幕分析结合解读。

来源：Glossary — Unresponsive Gesture

UI Freeze界面卡顿 / UI 冻结

定义：应用在用户操作过程中出现主线程长时间阻塞，无法及时处理输入，表现为界面「卡住」；应用未必被系统强制关闭（与 Crash 不同）。

检测与统计口径：SDK 在主线程上按固定间隔投递检测代码；若两次执行之间的间隔异常拉长，且超过约 1 秒（官方 Glossary 表述），则判定发生一次 UI Freeze，并捕获当时的堆栈等信息。可在会话筛选器的 Events 下按 UI Freeze 过滤，并查看 UI Freeze log。

来源：Glossary — UI Freezes、 UI Freeze's Events（与 Crash 对比说明）

Crash / Crashes崩溃

定义：未处理的异常或信号导致应用意外退出，用户需重新打开应用才能继续使用。

统计口径：按崩溃类型、次数、影响用户数/会话数聚合；与 UI Freeze（不必然杀进程）区分统计。

来源：Glossary — Crashes、 UI Freeze vs Crash

Exception异常

定义：在移动应用开发语境下，指偏离预期流程的未预见错误或非常态事件，可能导致应用异常表现、崩溃或非预期结果；通常通过捕获与处理机制来保障稳定性与体验。

说明：与 Crash（未处理导致进程退出）及 Handled exceptions（已捕获并记录的异常）在阅读 Issues、崩溃报表时经常一起出现；具体是否计入某张报表以产品界面与套餐为准。

来源：Glossary — Exception

时间与上传

Device Time设备时间

定义：会话开始时用户设备上的本地时间。

统计口径：依赖终端系统时钟；若用户将设备时间调快/调慢，可能出现与常识不符的时间显示，可结合 Uploading Time 判断。

来源：Glossary — Device Time、 Session FAQs — 未来时间

Recording Time录制时间（UTC）

定义：SDK 记录该会话时对应的 UTC 时间。

统计口径：用于与设备本地时间对照；便于跨时区对齐。

来源：Glossary — Recording Time

Uploading Time上传时间（UTC）

定义：会话数据上传到 UXCam 服务器的时间（UTC）。

统计口径：一般在应用进入后台后数秒内完成上传；排查「看不到最新数据」时可优先看该时间。

来源：Glossary — Uploading Time、 Session FAQs — 上传时机

漏斗与转化

Conversion rate / Total conversions转化率 / 总转化次数

定义：完成漏斗全部步骤的会话或用户占比；以及完成全路径的绝对次数。

统计口径：受漏斗配置影响：步骤为 This order 时需按顺序完成（中间可有其他行为）；Any order 时只需集齐各步不要求顺序。统计对象可选 Session count 或 User count。

来源：How to analyze a Funnel、 How to create a Funnel

Drop-off / Micro conversion流失 / 步骤级微转化

定义：漏斗中在某一相邻步骤之间未完成下一步的会话/用户比例称为流失；每一步的到达率可视为微转化。

统计口径：在漏斗图上可点击条带查看对应会话录屏；仅有序漏斗支持「平均全程转化耗时」等指标（两步骤漏斗可分析单段耗时）。

来源：How to analyze a Funnel

Conversion window转化窗口

定义：从进入漏斗到完成末步所允许的最长时间范围。

统计口径：按 Session 统计时，默认须在同一会话内完成所有步骤；按 User 统计时可设「7 天内」「24 小时内」等跨会话窗口。

来源：How to create a Funnel — conversion window

留存与队列

Cohort队列 / 同期群

定义：在同一时间段内完成了「起始行为」的用户集合，用于观察他们后续是否重复「回访行为」。

统计口径：常用按日/周/月/季切队列；可与用户属性、设备属性再做交叉分组。

来源：How does UXCam define Cohorts

Start action / Return action起始行为 / 回访行为

定义：留存报表中用于划定队列起点的行为（如注册、首访）；以及用来衡量「是否回来」的重复行为（如下单、打开核心页）。

统计口径：可配置多个行为及「最少发生次数」；回访间隔可选日/周/月/季，并区分 In（Bounded） 与 In or After（Unbounded） 留存读法。

来源：How to set up a Retention Report

筛选、分组与问题

Filter / Group by筛选 / 分组

定义：Filter 限定纳入分析的数据子集；Group by 将结果按维度拆分对比（时间、设备属性、用户属性等）。

统计口径：看板报表中分组最多两个维度，且官方说明部分场景不支持按小时分组；漏斗、热力图、会话列表等处的筛选条件各自独立作用于当前视图。

来源：Building Dashboards and Reports

Issues / Handled exceptions问题 / 已处理异常

定义：Issues 分析区通常汇总崩溃（Crash）、UI Freeze 等体验与稳定性问题；部分套餐/平台在报表构建中可选择 Handled exceptions 等数据点。

统计口径：按问题类型、版本、机型聚合影响面；可下钻到具体会话与堆栈。

来源：Crash and UI Freeze Analytics

日期范围与导出

Absolute / Relative date range绝对 / 相对日期范围

定义：Absolute 固定起止日期；Relative 随「今天」滚动而保持相对长度（如最近 7 天）。

统计口径：各报表、漏斗、会话列表的日期选择器均遵循当前视图所选类型。

来源：Glossary — Absolute / Relative date range

Data export数据导出

定义：将看板或列表中的数据导出为文件（如 CSV、图表为图片/PDF 等，以界面为准）。

统计口径：导出内容反映导出时刻的查询条件与权限范围内的数据快照。

来源：Product Analytics — Export Data

账户、密钥与隐私

API Key / App ID / App keyAPI 密钥 / 应用标识

定义：API Key 由你在账户中设定、在 API 调用中作为凭据使用的密钥，用于程序化访问应用数据；App ID 为应用在 API 中的唯一标识。集成/SDK 中常见 App key（与账户绑定，错配会导致数据无法上报）。官方提醒：API Key 与 App key 可在不登录控制台的情况下访问 UXCam 服务器上的数据，须妥善保管。

统计口径：与指标计算无直接关系，但错误配置会造成「无会话/无数据」表象。

来源：Glossary — API Key / App ID、 Session FAQs — account-key

Occlusion / Red screens遮挡 / 红屏

定义：开发者在集成时对敏感界面做遮挡处理；录屏中对应区域可能显示为红块以保护隐私。

统计口径：不代表崩溃；与合规配置相关。

来源：Session FAQs — red squares

IP address storageIP 地址处理

定义：官方说明不在数据库中长期存储用户 IP；首次请求时可用 IP 推断城市/国家后丢弃。

统计口径：Top country 等与地理相关的指标依赖该推断流程，与「是否存 IP」不是同一概念。

来源：Session FAQs — IP addresses

技术词汇（排障时常见）

SDK软件开发套件

定义：软件开发套件（Software Development Kit）：工具、库、文档等的集合，为开发应用提供代码示例、API、实用程序等资源。UXCam SDK 即嵌入应用的采集与分析组件及相关文档。

说明：会话起止、UI Freeze 检测、数据上传等行为均由 SDK 在客户端完成。

来源：Glossary — SDK

DPI屏幕点密度

定义：Dots per inch，设备屏幕的每英寸点数，用于描述显示精细度。

说明：出现在设备/会话元数据或统计维度中时，用于区分不同像素密度的机型；与业务漏斗无直接换算关系。

来源：Glossary — DPI

View Controller视图控制器（iOS）

定义：负责管理、协调应用某一屏界面呈现的架构组件：组织视觉元素、处理用户交互，并常封装该屏相关业务逻辑（如登录页的 LoginViewController）。

说明：UXCam 自动识别屏幕名时常显示此类类名；与业务「页面」可读性不一致时，见「页面命名」Tab 或 Screen Tagging API。

来源：Glossary — View Controller

Fragments片段（Android）

定义：可复用的模块化 UI 片段，常嵌在 Activity 内以搭建灵活、响应式布局；为界面中自包含的一块区域。

说明：与 Activity 组合出现时，Screens 可能显示 Fragment 相关名称。

来源：Glossary — Fragments

MainActivity主 Activity（Android）

定义：Android 应用中作为主入口的 Java/Kotlin 类：负责初始化主界面、处理用户交互并协调整体导航与行为，在启动时加载布局并拉起核心能力。

说明：常作为用户打开应用后首个或核心容器屏出现在 Screens 列表中。

来源：Glossary — MainActivity

Reverse flow逆向用户流

定义：在用户流分析中倒置屏幕顺序，用于从「退出前」反查用户此前经过的路径。

统计口径：不改变单条会话内事件顺序，仅改变可视化方向。

来源：Glossary — Reverse flow

Top event vs Top events单用户热门事件 vs 全应用热门事件

定义：Top event（单数）常指某用户在所选时段内触发最多的一个事件；Top events（复数）常指全应用范围内触发次数最多的若干事件。

统计口径：具体展示位置（用户详情 vs 用量概览）决定含义，阅读图表标题与轴标签确认。

来源：Glossary — Top event / Top events

报表与看板

Dashboards看板 / 仪表盘

定义：可自定义的集中视图，用于监控应用 KPI、分析体验并产出可行动结论。

统计口径：看板内各图表仍遵循底层指标（会话、事件、用户等）各自的统计定义；看板本身为展示与聚合容器。

来源：Glossary — Dashboards

Top events / Top app version / Top week day / Top country / Top devices热门事件 / 常用版本 / 高峰星期几 / 主要国家 / 常见设备尺寸

定义：用量与体验概览中的「Top」类指标，表示在选定时间窗口内出现频率最高或占比最大的维度取值。

统计口径：在选定周期内按事件次数、版本分布、会话发生星期、会话来源国家（据官方说明与 IP 推断地理）、设备尺寸等聚合排序；具体以 Usage & UX Statistics 文档为准。

来源：Glossary — Top events 等、 Usage & UX Statistics
