产品需求协作 SOP

统一“反馈 → 需求 → 版本”的流转规则：让每条反馈都有去向、每个需求都有评估、每个版本都有清晰的上线清单。适用于产品部及协作方（业务、运营、研发、测试）。

三表定位 整体流程 分阶段规范 字段映射 多端与版本 状态交接 角色职责 优化建议

03分阶段操作规范

三个阶段，对应三张表的录入与流转动作。

阶段 1 · 反馈收集与初筛

入口规则：所有反馈一律先进「反馈收集表」，填清反馈内容、提交人、来源页面 / 版本，必要时附图；不在群聊口头流转。

转产品需求池

需通过产品方案（页面 / 流程 / 能力 / 机制 / 配置后台）解决的真实、重复出现的问题。

转测试复测

紧急 Bug、线上故障，交测试复测并安排迭代节奏，直接进版本迭代表（类型＝紧急 Bug）。

直接处理 / 不处理

答疑、个案、对功能的误解、商务 / 法务 / 财务等非产品职责、方向不清的伪需求；在“备注”写明结论。

  • 入池口径：满足“对产品做改动以解决真实且会反复出现的用户 / 业务问题”才转需求池；纯数据 / 运营任务、Bug、个案、使用疑问、非产品职责、情绪化表达均不入池。
  • 可回溯：完成分流后标注去向，转入需求池 / 版本迭代表的反馈在“备注”回填关联需求名称。

阶段 2 · 需求池评估与排期

  • 需求描述：交付物式的一句话标题，不写症状。
  • 需求背景 / 原话：需求方原话 + 来源，不掺分析。
  • 需求评估备注：真实问题 / 价值与契合 / 优先级理由 / 风险与开放问题。
  • 其余字段：产品、所属功能、需求方、前后端工期 PD、前置依赖据实填写；进展状态新需求默认“未启动”。

优先级判定（P0–P3）

级别含义典型场景
P0上线阻断 / 核心链路 / 合规 / 过审 / 付费履约登录注册、Onboarding 断链、支付、崩溃级体验、过审与合规、已付费权益
P1显著拉动核心指标但不阻断上线明显提升激活 / 留存 / 转化 / 核心学习体验
P2收益明确但不紧急的体验优化 / 增值中长期价值、非核心路径优化、细分增值
P3探索 / 锦上添花 / 成本高收益不确定 / 远期重养成、复杂建模、与当前定位偏离

版本确定机制（两步）

① 产品经理初判

综合版本定位 + 需求优先级 + 资源评估，给出建议上线版本。

② 每周三 · 产品周会拍板

与辰光老师及产品团队共同评审，确定最终上线版本，回填需求池「上线版本」，随后流转到版本迭代表。

需求池进展状态流转：

未启动→ 产品设计中→ 待评审→ UI设计中→ 排期中→ （建记录到版本迭代表）→ 开发中→ 已完成

阶段 3 · 版本迭代执行

四类信息来源对应「类型」字段：

来源类型字段取值进入方式
产品需求产品需求由需求池“排期中”确定版本后流转
紧急 Bug紧急Bug由反馈收集表经测试复测后流转
技术自研优化研发优化研发直接提报
线上体验问题体验优化线上发现 / 反馈收集表流转
  • 录入：需求名称、所属产品板块、类型、优先级、负责人 / 研发 / 测试、提测时间、预计上线、PRD / UI / 技术文档、App版本、需求简介。
  • 用法：按“App版本”筛选，即可看到当前版本要上线的全部内容（产品需求 + 紧急 Bug + 研发优化 + 体验优化）。

项目状态流转：

需求沟通中→ PRD编写中→ PRD已评审→ UI设计中→ 开发中→ 测试中→ 已上线· 挂起（异常）

04需求池 → 版本迭代表 字段映射

交接时按此对应填写，避免两表口径漂移。

产品需求池版本迭代表说明
需求描述需求名称直接沿用
所属功能所属产品板块语义对应，按版本迭代表选项取值
优先级（P0–P3）优先级（P0–P2）P3 一般不进版本；若进版本按 P2 或重估
上线版本（V1.3.0）App版本（1.3.0…）命名保持一一对应
需求方需求方直接沿用
后台版本后台版本非 App 端用；命名“系统-批次”，App 需求留空
需求评估备注 / 背景需求简介 / 需求BRD提炼为执行视角简介
前 / 后端工期 PD、前置依赖提测时间 / 预计上线 / 研发 / 测试执行阶段补全排期信息

05多产品线与版本号

App 端之外还有管理后台 / 业务后台等需求——统一一个池、一张迭代表容纳，用「产品」区分端，版本号按端分轨。

一个需求池

所有端的需求都进同一个池，用已有的「产品」字段（用户端 / 官网 / 配套后台 / 单词管台）区分，跨端依赖一眼可见。

一张迭代表

所有端的执行也在同一张迭代表，用「产品板块 / 端」过滤；版本维度按端拆字段，互不打架。

版本号分轨

App 端走离散版本号 + 商店审核；后台持续部署，按“系统-批次”走，不套 App 版本号。

端版本字段口径与命名
App 用户端上线版本 / App版本离散版本号 1.3.0 / 1.3.2，绑应用商店审核发版
配套后台 / 业务后台 / 单词管台 / 官网后台版本持续部署，按“系统-批次”命名：如 配套后台-0703、单词管台-2026W28
  • 填写规则：App 需求填 App版本、后台版本留空；后台需求填后台版本、App版本留空——用哪个端就填哪个版本字段。
  • 命名规则：后台版本统一“系统-批次”，一个字段即可区分是哪个后台的哪一批，不为每个后台单独建字段。
  • 跨端联动：App 版本依赖后台改造时（如会员功能需后台先支持），用「前置依赖 / 父记录」关联两条记录，保证一起上线，版本号各记各的。

已落地：需求池与迭代表均已新增「后台版本」单选字段；后续如需把版本当实体管理（记发布日期 / 负责人 / 含哪些需求），可再升级为独立「版本登记表」+ 关联字段。

06状态与交接对齐

交接触发点：需求池状态到「排期中」且已确定 App 版本时，在版本迭代表创建对应记录（类型＝产品需求），并回填需求池「上线版本」；此后执行进度以版本迭代表「项目状态」为准，需求池在上线后置「已完成」。

  • 优先级口径：需求池 P0–P3、版本迭代表 P0–P2；P3 需求若进入版本须重新定级。
  • 版本号口径：需求池「上线版本」与版本迭代表「App版本」保持对应，避免一个需求两处版本号不一致。

08字段优化建议（反馈收集表）

⚠ 当前「处理状态」仅“已完成 / 暂不处理”，体现不了分流去向

  • 方案 A：扩充「处理状态」选项 —— 新增“转需求池”“转测试”“处理中”“不处理”。
  • 方案 B：新增「流转去向」单选（需求池 / 测试 / 直接处理 / 不处理）+「关联需求」文本，记录对应需求名称，打通三表回溯。
